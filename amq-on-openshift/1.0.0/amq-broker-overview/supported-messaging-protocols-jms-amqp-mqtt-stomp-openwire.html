Supported Messaging Protocols (JMS, AMQP, MQTT, STOMP, OpenWire)

AMQ Broker, built on Apache ActiveMQ Artemis, is designed to be highly versatile and interoperable. It achieves this by supporting a wide array of industry-standard messaging protocols. This multi-protocol support allows diverse client applications, written in different languages and running on various platforms, to communicate seamlessly with the broker.

The ability to speak multiple "messaging languages" means you can integrate new and legacy systems, connect lightweight IoT devices, enable real-time web applications, and build robust enterprise solutions all with a single AMQ Broker deployment.

Artemis Core Protocol

Red Hat AMQ uses a native, high-performance, JMS-agnostic Core protocol designed for maximum speed, low latency, and advanced broker features beyond standard JMS. It supports client-side compression, large messages, and acts as the underlying engine for other protocols like AMQP and MQTT, offering a specialized API for direct, efficient communication.

Key Aspects of the ActiveMQ Artemis Core Protocol

  • Native & Optimized: It is the native wire format of Artemis, built for speed and efficiency.

  • JMS-Agnostic Core API: You can use the Core API directly for advanced features not available in standard JMS, or use it to implement JMS.

  • Feature-Rich: Supports advanced messaging capabilities, including large message compression, clustering, and security.

  • Client Flexibility: While it has its own dedicated API, JMS applications can also use the AMQ Core Protocol JMS client, which maps JMS calls to Core protocol operations.

Refer the upstream documentation for more information.

Advanced Message Queuing Protocol (AMQP)

The Advanced Message Queuing Protocol (AMQP) is an open, wire-level protocol for message orientation, queuing, routing, reliability, and security. Unlike JMS, which is an API specification for Java, AMQP is a network protocol that enables interoperable messaging between different programming languages and platforms. This means an application written in Python using an AMQP client can seamlessly exchange messages with an application written in Java using an AMQP client, both communicating with the same AMQ Broker.

AMQ Broker fully supports AMQP 1.0, which is a standardized protocol offering high performance and advanced messaging capabilities. AMQP 1.0 focuses on the reliable delivery of messages and provides sophisticated features for message routing, acknowledgments, and flow control.

Key Features of AMQP with AMQ Broker

  • Interoperability: Being a wire-level protocol, AMQP enables client applications written in various languages (Java, Python, C++, Node.js, etc.) to communicate effortlessly.

  • Reliable Messaging: Provides robust mechanisms for message delivery guarantees, including transactional semantics and various acknowledgment modes.

  • Flexible Routing: Supports complex routing scenarios, allowing messages to be directed to specific queues or topic subscriptions based on properties or content.

  • Security: Integrates with TLS/SSL for encrypted communication.

  • Flow Control: Mechanisms to prevent producers from overwhelming consumers or the broker.

  • Open Standard: Governed by ISO/IEC 19464, ensuring wide adoption and long-term stability.

AMQP is particularly well-suited for cloud-based messaging, IoT solutions where diverse devices need to communicate, and microservices architectures requiring language-agnostic message exchange.

Message Queuing Telemetry Transport (MQTT)

MQTT is a lightweight, publish-subscribe messaging protocol designed for constrained devices and low-bandwidth, high-latency, or unreliable networks. It’s become the de-facto standard for Internet of Things (IoT) applications due to its efficiency and simplicity.

AMQ Broker provides robust support for MQTT, making it an excellent hub for connecting a vast array of IoT devices, sensors, and mobile applications.

Key Features of MQTT with AMQ Broker

  • Lightweight: Minimal message overhead and small code footprint, ideal for resource-constrained devices.

  • Publish/Subscribe Model: Decouples senders (publishers) from receivers (subscribers), allowing for flexible and scalable architectures.

  • Quality of Service (QoS): Offers three levels of message delivery guarantees:

  • QoS 0 (At most once): Fire and forget.

  • QoS 1 (At least once): Ensures the message arrives but might be duplicated.

  • QoS 2 (Exactly once): Guarantees the message arrives once and only once.

  • Persistent Sessions: Allows clients to reconnect and resume their session, receiving messages that were published while they were offline.

  • Last Will and Testament: A feature where a client can specify a message to be sent by the broker to subscribers if the client disconnects unexpectedly.

  • Security: Supports TLS/SSL for secure communication.

MQTT is the protocol of choice for smart devices, industrial IoT, telemetry, and mobile messaging where network conditions can be challenging and resource consumption needs to be minimized.

Simple Text-Oriented Message Protocol (STOMP)

STOMP is a simple, text-based messaging protocol designed for working with message brokers. It provides an interoperable wire format that allows STOMP clients to communicate with any STOMP-compatible message broker. Its simplicity makes it particularly popular for scripting languages, web browsers (via WebSockets), and lightweight applications.

AMQ Broker includes native STOMP support, enabling web-based and simple clients to interact with queues and topics effectively.

Key Features of STOMP with AMQ Broker

  • Simplicity: Easy to implement and use, often requiring less complex client libraries compared to AMQP or JMS.

  • Text-Oriented: Messages are typically text-based, making it human-readable and easy to debug.

  • Frame-Based: Uses a frame-based structure (command, headers, body) for sending and receiving messages.

  • Web-Friendly: Commonly used over WebSockets, allowing real-time messaging directly from web browsers to the broker.

  • Publish/Subscribe and Point-to-Point: Supports both messaging paradigms.

  • Broad Client Support: Libraries available for almost every modern language and platform.

STOMP is an excellent choice for web applications needing real-time updates, browser-based chat applications, or scripting environments where rapid development and simplicity are prioritized.

OpenWire (Deprecated)

OpenWire is a high-performance, compact, and extensible wire format specifically developed for Apache ActiveMQ. It’s the native protocol primarily used by legacy Apache ActiveMQ clients and provides an optimized way for Java clients to communicate with ActiveMQ-based brokers. While AMQ Broker is based on Apache ActiveMQ Artemis (a next-generation broker), it maintains compatibility with OpenWire to support existing ActiveMQ client applications.

This compatibility is crucial for organizations migrating from Apache ActiveMQ 5.x to Red Hat AMQ Broker, allowing them to leverage the new broker without immediately rewriting their client applications.

Key Features of OpenWire with AMQ Broker

  • High Performance: Designed for efficient message transfer, particularly within Java ecosystems.

  • Optimized for Java: Provides a rich set of features tailored for Java clients, including object serialization.

  • Backward Compatibility: Enables older Apache ActiveMQ clients to connect to AMQ Broker, facilitating migrations.

  • Extensible: Allows for easy addition of new commands and features to the protocol.

  • Client-Side Features: Supports advanced client-side features like failover, redelivery policies, and consumer prefetch.

OpenWire is essential for scenarios where existing Apache ActiveMQ 5.x Java clients need to connect to an AMQ Broker instance, providing a smooth transition path and preserving existing client investments. This protocol is deprecated in the AMQ Broker and will be removed in one of the next major releases.
